09 · ADR — Architecture Decision Records (3건)
ADR-001 ACCEPTED
저장소: Hive vs SQLite
SQLite는 관계형 스키마 마이그레이션 관리가 필요하고, 직렬화 코드를 수동으로 작성해야 한다. 앱은 서버 없이 단독 동작하며 간단한 Key-Value 저장이 주를 이룬다.
✅ 결정: Hive 선택 — TypeAdapter 코드 자동 생성, NoSQL Key-Value로 보일러플레이트 최소화, 오프라인 특화 빠른 읽기
트레이드오프: 복잡한 JOIN 쿼리 불가 → 앱 특성상 조인 불필요해 영향 없음
ADR-002 ACCEPTED
상태관리: Riverpod vs Provider
Provider는 BuildContext 의존성과 런타임 타입 오류 위험이 있다. Riverpod은 컴파일 타임에 타입을 검증하고 코드 생성으로 Provider 선언을 자동화한다.
✅ 결정: Riverpod 2.x 선택 — @riverpod 어노테이션 코드 생성, 컴파일 타임 안전성, Context 없는 Provider 접근
트레이드오프: build_runner 학습 비용 → 초기 설정 후 생산성 대폭 향상
ADR-003 ACCEPTED
아키텍처: Layered+MVVM vs Clean Architecture
Clean Architecture는 UseCase·Entity·Repository가 완전히 분리되어 팀·대규모 프로젝트에 적합하나 1인 1주 개발에는 추상화 레이어가 과도하다.
✅ 결정: Layered + MVVM 선택 — Presentation / Domain / Data / Core 4계층, 단방향 의존성, 최소 추상화
트레이드오프: 향후 팀 확장 시 Clean Architecture로 마이그레이션 필요 가능성 존재
