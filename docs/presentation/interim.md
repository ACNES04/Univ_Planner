# 01. 프로젝트 개요

- 프로젝트명: 소비 반성 일기 (지출 챌린지)
- 목표: 충동 소비를 기록하고, 감정(만족/후회)과 반성 문장을 함께 남겨 소비 습관을 개선
- 발표 범위: 10~12주차 산출물(기획 + 아키텍처 + 동작 데모)

---

# 02. 문제 정의

## 누구의 문제인가?
- 대학생/사회초년생처럼 월 예산 관리가 어려운 사용자

## 어떤 문제인가?
- 지출 직후에는 금액만 기억하고, 왜 샀는지 감정/맥락을 놓침
- 같은 후회 소비가 반복되지만 회고 데이터가 없어 개선이 어려움

## 왜 지금 푸는가?
- 소비 습관은 작은 기록 루틴으로도 빠르게 개선 가능
- 중간발표 시점에서 동작 가능한 MVP를 통해 사용자 가치 검증 가능

---

# 03. 핵심 기능(MVP)

- 지출 기록 등록: 지출명, 금액, 감정(만족/후회), 반성 한 줄
- 리스트 조회: 최신 순 정렬, 감정 상태에 따른 시각화
- 챌린지 카드: "후회 없는 소비 챌린지" 진행 문구 표시
- 인메모리 더미 데이터 3건 내장: 발표장에서 즉시 시연 가능

---

# 04. 기술 스택 및 ADR

| 구분 | 선택 기술 | 선택 이유 | ADR |
|---|---|---|---|
| UI 프레임워크 | Flutter | 단일 코드베이스로 모바일 시연 용이 | ADR-0001 |
| 상태 관리 | Riverpod(StateNotifier) | 의존성 주입 + 테스트 친화적 구조 | ADR-0001 |
| 데이터 소스 | In-Memory Dummy Data | 발표장 네트워크/DB 변수 제거, 즉시 재현 가능 | ADR-0002 |
| 아키텍처 | Presentation → Application → Domain → Data | 책임 분리, 코드 검수 대응 용이 | ADR-0001 |

- ADR-0001: Clean/Layered Architecture + Riverpod 채택
- ADR-0002: 중간 발표 버전은 인메모리 저장소 사용

---

# 05. 아키텍처 다이어그램

~~~mermaid
flowchart TD
    P[Presentation\nHomeScreen] --> A[Application\nConsumptionViewModel]
    A --> Dm[Domain\nConsumptionItem]
    A --> Dt[Data\nConsumptionRepository]
    Dt --> Mem[(In-Memory Dummy List)]
~~~

- 단방향 흐름 유지: UI는 ViewModel만 호출
- 데이터 접근 세부 구현은 Data 레이어에 은닉

---

# 06. 데모 시연 시나리오

1. 앱 실행 직후 더미 데이터 3건이 목록에 보임
2. 하단 "소비 기록하기" 버튼 클릭
3. 지출명/금액/감정/반성 입력 후 "등록하기"
4. 목록에 즉시 반영되는 상태 변화를 시연
5. 감정별 카드 색상 차이(후회=붉은 톤, 만족=푸른 톤) 강조

- 전체 진척률: 약 50%

---

# 08. 코드 검수 대응 포인트

- 레이어 책임 분리 명확:
  - Presentation: 화면 렌더링/입력 감지
  - Application: 유스케이스 조율/상태 업데이트
  - Domain: 순수 엔티티
  - Data: 데이터 출처 은닉
- 발표 리스크 최소화:
  - 네트워크/DB 의존 없음
  - 더미 데이터 포함으로 cold-start 즉시 시연 가능
- 유지보수성:
  - Data 구현체만 바꾸면 영속 저장소로 확장 가능

---

# 09. 다음 단계(최종발표까지)

- 인메모리 → 로컬 DB(Hive/SQLite) 전환
- 수정/삭제/통계(주간 후회율) 기능 확장
- 테스트 케이스 확대(위젯/유스케이스)
- 데모 영상과 발표 스크립트 동기화

## 2분 발표 타이머 스크립트 (120초)

- 00:00~00:30 (기획 의도 30초)
  - 안녕하세요. 저희 팀 주제는 소비 반성 일기, 즉 지출 챌린지 앱입니다. 문제는 지출 기록이 금액 중심으로 끝나서 같은 실수를 반복한다는 점입니다. 그래서 저희는 지출명, 금액뿐 아니라 감정(만족/후회)과 반성 한 줄을 함께 기록해 소비 습관을 스스로 점검할 수 있게 설계했습니다.
- 00:30~00:55 (아키텍처 25초)
  - 구조는 교수님 가이드대로 Presentation, Application, Domain, Data 4계층 단방향 흐름을 지켰습니다. 화면은 ViewModel만 호출하고, ViewModel은 Repository를 통해 데이터를 다룹니다. 이렇게 분리해서 코드 검수 시 책임이 명확하고, 나중에 저장소를 바꿔도 영향 범위가 작습니다.
- 00:55~01:35 (데모 시연 멘트 40초)
  - 지금 데모를 보시면 앱 실행 직후 더미 데이터 3건이 먼저 보여서 바로 시연이 가능합니다. 하단 소비 기록하기 버튼을 누르면 입력 폼이 뜨고, 지출명/금액/감정/반성 한 줄을 입력한 뒤 등록하기를 누르면 리스트에 즉시 반영됩니다. 후회는 붉은 카드, 만족은 푸른 카드로 구분되어서 상태를 직관적으로 확인할 수 있습니다.
- 01:35~02:00 (WBS 마무리 25초)
  - 현재 12주차 기준 전체 진척도는 50%입니다. 기획과 구조 설계, MVP 핵심 기능은 완료했고, 다음 단계는 테스트 강화와 로컬 DB 전환, 최종 리허설입니다. 중간발표 이후에는 안정성과 완성도를 높여 최종 결과물로 발전시키겠습니다.
